# 第一部分：鋼板「月內多週價格」改取最後一週

## 1.1 問題

工程會鋼板 PDF 在 **民國 103/01 ～ 108/12** 期間，一份月報內含當月四次週調查（如 108/12 含 12/06、12/13、12/20、12/25）。程式目前把同月、同品項、同地區的多週價格**取算術平均**當作當月值，會算出一個實際上任何一週都沒出現過的數字。

**實例（108/12 鋼板 A36）**：四週為 22,000 / 20,300 / 20,300 / 20,300，平均 = **20,725**（現行 CSV 值）。但當月最後一週（12/25）實際為 **20,300**。

## 1.2 需求

**當一個月內有多筆週調查時，以「最後一週（日期最新）」的價格為當月值，不取平均。** 單週月份（含 109/04 改版後所有月份）行為不變。

## 1.3 修改內容（唯一改動）

檔案 `src/consolidate_pdf.py`，`main()` 內的聚合區塊（約第 290 行）：

**修改前**
```python
agg = df.groupby(keys, as_index=False, dropna=False).agg(
    price=("price", "mean"),          # 取平均
    price_min=("price_min", "min"),
    price_max=("price_max", "max"),
    source_file=("source_file", "first"))
```

**修改後**
```python
agg = df.groupby(keys, as_index=False, dropna=False).agg(
    price=("price", "last"),          # 取最後一週
    price_min=("price_min", "min"),
    price_max=("price_max", "max"),
    source_file=("source_file", "first"))
```

即把 `price` 的聚合函式由 `"mean"` 改為 `"last"`，其餘皆不動。

## 1.4 為何這樣就足夠（不需其他改動）

- `parse_pdf()` 讀取一份 PDF 時，是依 `page.extract_tables()` 回傳的順序處理各週表格；該順序為頁面上由上而下，**即週調查的時間先後**（例如 108/12 依序為 12/06 → 12/13 → 12/20 → 12/25）。
- 各週資料列依此順序被附加進清單，`main()` 建成 DataFrame 後**未經重新排序**即進行 `groupby`。因此每一組（期別×品項×地區…）中，**最後一筆即為最後一週**。
- `groupby(...).agg("last")` 取每組最後一筆，故直接得到「最後一週」的值。無需記錄週別日期，也無需改動 `parse_pdf()`。

> 附帶說明：`"last"` 會略過空值（取最後一個有效值）。若最後一週該格恰為空，會自動退回到最近一個有值的週，屬合理行為。

# 第二部分：修正後即時同步 GitHub，供 Power BI 驗證

## 2.1 背景

Power BI 的資料來源是 GitHub 上的 raw 網址（`raw.githubusercontent.com/.../data/history/material_price_history.csv`），並非本機檔案。若修正後的 CSV 只更新在本機、未推上 GitHub，Power BI 讀到的仍是**舊資料**，將無法正確檢視這次變動的實際影響。

## 2.2 要求

1. 程式修改完成後，需**對全部歷史 PDF 重新執行一次 `consolidate_pdf.py`**，重新產出 `data/history/material_price_history.csv`（因為改動會回溯影響過去多年的月份，不是只影響最新一期）。
2. 重算後的 CSV 必須**立即 commit 並 push 到 GitHub `main` 分支**，不可只留在本機。
3. push 完成後，請於 Power BI 服務端**手動觸發一次資料重新整理**，並確認抓到的是修正後的值（可用 108-12 鋼板 A36 是否已由 20,725 變為 20,300 作為確認點）。

## 2.3 目的

確保在 Power BI 上檢視「這次修正造成哪些變化」時，看到的是正確、已同步的資料，而非新舊混雜。

> 附註：平時的 `scripts/run_pipeline.ps1` 已包含自動 commit/push；但本次是「一次性全歷史重算」，請特別確認重算結果確實推上 GitHub，不要遺漏。

# 第四部分：雲端執行歷程紀錄（可長期回溯、且顧及安全）

## 4.1 目標

在 GitHub 上保留一份**每次排程執行都會累積**的執行歷程，讓我們日後隨時能在雲端查詢「這套自動更新有沒有持續、正常地跑」，而不只看得到最近一次或當月的結果。同時**不得**把可能夾帶敏感資訊（本機路徑、認證權杖、原始錯誤輸出）的內容上傳。

## 4.2 設計：兩層紀錄，各司其職

| 紀錄 | 檔案 | 是否上傳 GitHub | 內容 | 用途 |
|---|---|---|---|---|
| 本機詳細日誌 | `run.log`（維持現狀，`.gitignore` 保留忽略） | 否，只留本機 | 完整輸出、git/python 訊息、錯誤堆疊 | 在該電腦上除錯用 |
| 雲端精簡歷程（新增） | `data/run_history.csv` | 是，每次執行追加一列並 push | 僅固定幾個安全欄位 | 隨時在 GitHub 回溯查詢 |

要點：
- 雲端歷程檔放在 `data/` 底下，沿用現有的 `git add data`，無需改動上傳範圍。
- 為 **append-only（只追加、不覆寫）**，歷史永久累積。
- **每次執行都追加一列並上傳**——「有新資料／無新資料／失敗」三種情形皆記錄。因歷程檔每次都變動，故每次執行必留下一筆雲端紀錄，可據以判斷排程是否持續運作。

## 4.3 雲端歷程檔欄位（`data/run_history.csv`）

僅記錄下列由腳本明確賦值的欄位，**不寫入路徑、權杖或任何原始命令輸出**：

| 欄位 | 說明 | 範例 |
|---|---|---|
| `run_utc` | 執行時間（UTC，ISO 格式） | `2026-07-21T00:03:11.5Z` |
| `result` | 結果分類 | `new_data` / `no_new_data` / `fetch_failed` / `consolidate_failed` |
| `fetch_exit` | `fetch_pdf.py` 退出碼 | `0` / `1` / `2` |
| `latest_period` | 目前 CSV 內最新期別 | `115-06` |
| `rows_total` | 目前 CSV 總列數 | `14808` |

## 4.4 參考實作：改寫後的 `scripts/run_pipeline.ps1`

在原腳本基礎上重構退出邏輯，使「記錄＋上傳」在任何結果下都會執行（此改動一併涵蓋第二部分要求的自動 push）。

```powershell
# Local pipeline: fetch PDFs -> parse -> record run -> git push
# 本機詳細日誌 run.log（gitignored，不上傳）；雲端精簡歷程 data\run_history.csv（上傳）
$ErrorActionPreference = 'Continue'
$env:PYTHONIOENCODING = 'utf-8'
$env:PYTHONUTF8 = '1'
try { [Console]::OutputEncoding = [System.Text.Encoding]::UTF8 } catch {}

$repo = Split-Path -Parent $PSScriptRoot
if (-not (Test-Path $repo)) { Write-Error "repo not found: $repo"; exit 1 }
Set-Location $repo
$log = Join-Path $repo 'run.log'

function Log($m) { Add-Content -Path $log -Encoding UTF8 -Value (("{0}  {1}" -f (Get-Date -Format 'o'), $m)) }
function Run($file, $argList) {
    & $file @argList 2>&1 | ForEach-Object { Add-Content -Path $log -Encoding UTF8 -Value "$_" }
    return $LASTEXITCODE
}

Log '=== pipeline start ==='

# 1) 下載
$fetchCode = Run 'python' @('src\fetch_pdf.py')
Log "fetch_pdf exit = $fetchCode"

# 2) 決定結果；有新資料才整理。不提前 exit，確保後面一定會記錄並上傳。
$result = ''
if     ($fetchCode -eq 1) { $result = 'fetch_failed' }
elseif ($fetchCode -eq 2) { $result = 'no_new_data' }
elseif ($fetchCode -eq 0) {
    if ((Run 'python' @('src\consolidate_pdf.py')) -ne 0) { $result = 'consolidate_failed' }
    else { $result = 'new_data' }
}
else { $result = "fetch_unexpected_$fetchCode" }
Log "result = $result"

# 3) 計算安全欄位（只讀公開資料，不含任何敏感內容）
$latestPeriod = ''; $rowsTotal = ''
$csvPath = Join-Path $repo 'data\history\material_price_history.csv'
if (Test-Path $csvPath) {
    try {
        $rows = Import-Csv -Path $csvPath
        $rowsTotal = $rows.Count
        $latestPeriod = ($rows | Sort-Object period | Select-Object -Last 1).period
    } catch { }
}

# 4) 追加一列到雲端執行歷程（append-only）
$histCsv = Join-Path $repo 'data\run_history.csv'
if (-not (Test-Path $histCsv)) {
    Add-Content -Path $histCsv -Encoding UTF8 -Value 'run_utc,result,fetch_exit,latest_period,rows_total'
}
$runUtc = (Get-Date).ToUniversalTime().ToString('o')
Add-Content -Path $histCsv -Encoding UTF8 -Value ("{0},{1},{2},{3},{4}" -f $runUtc, $result, $fetchCode, $latestPeriod, $rowsTotal)

# 5) 上傳（歷程檔每次都變動，故每次執行都會留下一筆雲端紀錄）
Run 'git' @('add', 'data') | Out-Null
if ((Run 'git' @('commit', '-m', ("run {0} {1}" -f (Get-Date -Format 'yyyy-MM-dd'), $result))) -ne 0) { Log 'commit FAILED'; exit 1 }
if ((Run 'git' @('push')) -ne 0) { Log 'push FAILED'; exit 1 }
Log 'pushed to GitHub'

# 6) 退出碼：失敗類回傳 1，其餘 0
if ($result -eq 'fetch_failed' -or $result -eq 'consolidate_failed' -or $result -like 'fetch_unexpected*') { exit 1 }
exit 0
```

## 4.5 其他須注意

- **確認 `data/run_history.csv` 未被 `.gitignore` 忽略**（目前僅忽略 `run.log` 與 `data/history/pdf/`，此新檔應可正常入庫；請實際確認一次）。
- **行為變化**：改動後**每次排程都會產生一個 commit**（連沒有新價格資料的週也是），此為刻意設計，用作「心跳」訊號；每年約多出數十個提交，對 git 無負擔。
- **附帶好處**：commit 訊息為 `run <日期> <result>`（如 `run 2026-07-28 no_new_data`），故 GitHub 提交列表本身即為一份可快速掃視的執行概覽。
- **push 失敗無法記錄於雲端**（因為正是 push 失敗），此情形只會留在本機 `run.log`，屬合理限制。

## 4.6 查詢方式

- **完整歷程**：在 GitHub 開啟 `data/run_history.csv`，可見自建立以來每一次執行的時間、結果、當時最新期別與列數。
- **快速概覽**：看 repo 的 commits 列表，一列即一次執行。
- **（選用）**：此 CSV 亦可另接入 Power BI 作為「資料更新健康度」小表監看。
